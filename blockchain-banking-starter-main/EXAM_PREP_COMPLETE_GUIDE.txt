4. BLOCKCHAIN EXPLAINED (EASY VERSION)
================================================================================

WHAT IS BLOCKCHAIN?
Think of blockchain like a notebook where you write transaction records:
- Each entry is permanent and cannot be edited
- Once you write something, it stays forever
- Everyone can see the record (transparent)
- If someone tries to change a record, everyone will know (immutable)

HOW BLOCKCHAIN WORKS IN THIS PROJECT:

Step 1: Transaction Occurs
User transfers ₹5000 from Account A to Account B
This transaction needs to be recorded. Here's the data:
  from: "ACC001"
  to: "ACC002"
  amount: 5000
  type: "transfer"
  timestamp: "2026-03-03T10:30:00Z"

Step 2: SHA-256 Hashing (Creating a Digital Fingerprint)
The system takes all transaction data and runs it through SHA-256 algorithm.
SHA-256 is like a special blender that:
- Takes input (transaction data)
- Mixes it in a complex way
- Outputs a unique 64-character "hash" (fingerprint)

Input:  "from=ACC001, to=ACC002, amount=5000..."
Output: "3e4a7c8f9f2d1b5e6c8a9d0f1e2c3b4a5e6f7g8h9i0j1k2l3m4n5o6p7q8r"

IMPORTANT: Same input always produces SAME hash
           Different input produces COMPLETELY different hash
           Even changing 1 character changes the entire hash

Step 3: Recording in Ledger (Blockchain Blocks)
The transaction + its hash is stored in a "block":
Block #42:
  blockNumber: 42
  hash: "3e4a7c8f..." (SHA-256 hash)
  previousHash: "7f2d1b5e..." (hash of previous block)
  timestamp: "2026-03-03T10:30:00Z"
  data: {from, to, amount, type, ...}

Step 4: Chain Linking
Each block contains the HASH of the PREVIOUS block
This creates a chain:
  Block #1 → Block #2 → Block #3 → Block #4 ...

If someone tries to change Block #2:
1. Block #2 hash changes
2. Block #3's previousHash no longer matches Block #2's hash
3. System detects tampering ❌

VERIFICATION PROCESS:
When you verify a transaction:
1. Recalculate the hash from stored data
2. Compare with the stored hash
3. If they match → Data is authentic ✓
4. If they don't match → Someone changed it ✗

IN THIS PROJECT:
- Blockchain records all transactions immutably
- Can verify any transaction's authenticity
- Data stored in memory (demo mode) or MongoDB (persistent mode)
- Uses SHA-256 hashing algorithm
- Creates audit trail for compliance

IMPORTANT: This is a DEMO blockchain, not a real one like Bitcoin/Ethereum
- Single node (not distributed)
- No mining or proof-of-work
- No cryptocurrency tokens
- Educational demonstration of blockchain concepts

✅ BLOCKCHAIN IS NOW WORKING & RECORDING TRANSACTIONS:
When a transaction is approved:
1. Backend calculates SHA-256 hash of transaction data
2. Block is created with: blockNumber, hash, previousHash, timestamp, data
3. Block is stored in MongoDB blockchain_blocks collection
4. Each new block links to previous block via previousHash
5. Creates immutable chain - tampering is immediately detectable

6. HOW EACH PAGE WORKS (DETAILED FLOW)
================================================================================

USER REGISTRATION & LOGIN FLOW:

Sign Up:
1. User navigates to /signup
2. Fills form: Name, Email, Phone, Address, DOB, Password, Account Type
3. Clicks "Sign Up"
4. Frontend sends POST request to /api/auth/register
5. Backend validates:
   - Email must be unique
   - Password must be strong
   - All required fields filled
6. Backend hashes password using bcrypt (10 salt rounds)
7. Creates new User document in MongoDB
8. Creates new Account document linked to user
9. Sends verification email (optional)
10. Redirects to login page

Login:
1. User enters Email and Password
2. Clicks "Login"
3. Frontend sends POST request to /api/auth/login
4. Backend:
   - Finds user by email
   - Compares entered password with hashed password using bcrypt
   - If match, generates JWT token
   - Returns token to frontend
5. Frontend stores token in localStorage
6. Frontend redirects to dashboard
7. All subsequent requests include token in headers

2FA (Two-Factor Authentication) FLOW:

Setup 2FA:
1. User goes to Settings, clicks "Enable 2FA"
2. Backend generates TOTP secret using Speakeasy
3. Creates QR code from secret
4. Frontend displays QR code
5. User scans with authenticator app (Google Authenticator, Authy)
6. User enters 6-digit code from app
7. Backend verifies code
8. Backend generates 10 backup codes
9. 2FA enabled successfully

Login with 2FA:
1. User enters email and password
2. System verifies credentials
3. If 2FA enabled, prompts for 6-digit code
4. User opens authenticator app, reads code
5. Backend verifies code against TOTP secret
6. Code valid for 30 seconds
7. If verified, JWT token issued
8. User logged in successfully

DEPOSIT FLOW:

1. User navigates to /deposit page
2. Fills form:
   - Amount (minimum ₹100)
   - Deposit method (bank_transfer or check)
   - Bank details (if bank_transfer)
   - Check details (if check)
   - Purpose (e.g., Salary, Investment)
3. Clicks "Request Deposit"
4. Frontend validates form
5. Sends POST to /api/deposits
6. Backend:
   - Creates Deposit record with status='pending'
   - Generates unique reference number
   - Stores all details in MongoDB
   - Sends email notification to user: "Deposit request received"
7. Deposit appears in user's transaction history as "Pending"
8. Admin sees in "Pending Deposits"

Admin Approves Deposit:
1. Admin navigates to Deposits Management
2. Sees pending deposits
3. Clicks "Approve" on a deposit
4. Backend:
   - Updates deposit status to 'completed'
   - Adds amount to user's account balance
   - Records blockchain hash
   - Sends email: "Deposit approved, ₹XXXX added"
5. User sees deposit now "Completed"

WITHDRAWAL FLOW:

1. User navigates to /withdraw page
2. Fills form:
   - Amount (cannot exceed balance)
   - Withdrawal method
3. Clicks "Request Withdrawal"
4. Frontend:
   - Validates amount ≤ balance
   - Sends POST to /api/withdrawals
5. Backend:
   - Creates Withdrawal record with status='pending'
   - Checks balance is sufficient
6. Gets approval from admin
7. Once approved:
   - Deducts amount from balance
   - Email sent: "Withdrawal approved"
8. Money transferred to user's bank

TRANSFER FLOW:

1. User navigates to /transfer page
2. Fills form:
   - Recipient account number
   - Amount
   - Purpose
3. Clicks "Send"
4. Frontend validates:
   - Amount > 0
   - Amount ≤ balance
   - Recipient exists
5. Sends POST to /api/transfers
6. Backend:
   - Creates Transfer record
   - Validates recipient account exists
   - Records transaction details
7. Transfer appears in both users' history
8. Recipient receives email notification

CREDIT SCORING FLOW:

1. User navigates to /credit-scoring
2. Page loads, checks if user has transactions
3. If NO transactions:
   - Page calls generateSampleData endpoint
   - Backend creates 5 deposits and 4 withdrawals
4. Page fetches user's transactions from database
5. Calls /api/credit/calculate endpoint
6. Backend calculates score based on:
   - Total deposits (income)
   - Total withdrawals (spending)
   - Deposit frequency
   - Withdrawal frequency
   - Transaction completion rate
   - Account age
7. Returns score (300-850) and grade (A+ to F)
8. Frontend displays:
   - Score and grade
   - 6-factor breakdown with percentages
   - Loan eligibility
   - Interest rate
   - Improvement tips

KYC VERIFICATION FLOW:

1. User navigates to /kyc page
2. Fills form:
   - Full Name (text field)
   - Address (text field)
   - Document Type (dropdown: Aadhar, PAN, Passport, Driver's License)
   - Document Number (text field)
   - Document File (upload PDF/JPG/PNG, max 5MB)
3. Frontend validates:
   - All fields required
   - File must be selected
   - File must be .pdf, .jpg, .jpeg, or .png (HTML accept attribute)

   ✓ File extension check: Allows only .pdf, .jpg, .jpeg, .png
   ✓ File size check: Rejects if > 5MB
   ✓ Invalid files rejected before reaching controller
   ✓ Error response: "Invalid file type" or "File size exceeds 5MB"

9. SECURITY FEATURES
================================================================================

AUTHENTICATION SECURITY:

1. Password Security:
   - Passwords are HASHED using bcrypt (one-way encryption)
   - 10 salt rounds (makes it very hard to crack)
   - Cannot be reversed to get original password
   - Example: "Password123" → "$2b$10$...(60 chars)..."
   
2. JWT Tokens:
   - Short-lived tokens (expires in few hours)
   - Tokens sent in Authorization header: "Bearer <token>"
   - Server validates token signature
   - Token contains user ID and role

3. Token Storage:
   - Stored in localStorage (browser storage)
   - Sent with every API request automatically
   - Cannot be accessed by other sites (same-origin policy)

TWO-FACTOR AUTHENTICATION (2FA):

1. TOTP (Time-based One-Time Password):
   - User has TOTP secret
   - Generates new 6-digit code every 30 seconds
   - Code is unique and time-based
   - Cannot be reused after 30 seconds

2. Backup Codes:
   - 10 recovery codes generated
   - Each code can be used once
   - For account recovery if phone lost
   - Must be stored safely

TRANSACTION SECURITY:

1. Blockchain Hashing:
   - All transactions get SHA-256 hash
   - Hash links to previous transaction
   - Tampering is immediately detectable
   - Creates immutable audit trail

2. Balance Verification:
   - System checks balance before withdrawal/transfer
   - Prevents overdrafts
   - Database transaction ensures atomicity

3. Fraud Detection:
   - Unusual patterns detected
   - Large transactions flagged
   - Multiple failed attempts blocked
   - Risk scoring on deposits

DATA PROTECTION:

1. Sensitive Data Fields:
   - Password fields excluded from queries
   - Token not returned to frontend in responses
   - Credit card data not stored
   - Personal info accessible only to owner

2. Encryption:
   - Connections use HTTPS (encrypted)
   - Data at rest in MongoDB (encrypted)
   - Sensitive fields hashed

AUTHORIZATION:

1. Role-Based Access Control:
   - Customer routes protected
   - Admin routes protected
   - Users can only see their own data
   - Admins see everything

2. Protected Routes:
   - <ProtectedRoute> wrapper checks token
   - If no token, redirects to login
   - If invalid role, shows 403 error

ACCOUNT SUSPENSION:

1. Prevents Fraud Users:
   - Suspicious accounts suspended
   - Suspended users cannot perform transactions
   - Cannot login during suspension
   - Automatic unsuspension possible

2. Audit Trail:
   - Log of who suspended whom
   - When suspension occurred
   - Reason for suspension
   - Unsuspension events

EMAIL SECURITY:

1. Verification Emails:
   - Users can verify email address
   - Prevents fake email registration
   - One-time verification links

2. Notifications:
   - Email alerts for sensitive actions
   - Transaction notifications
   - Suspension alerts
   - Helps detect unauthorized access

12. KEY CONCEPTS EXPLAINED (EASY VERSION)
================================================================================

CONCEPT 1: REST API
What: A way for frontend and backend to talk
How:
  Frontend sends request → Backend processes → Returns response
  Kind of like:
  "Hi backend, give me user info" → Backend: "Here it is"
  "Hi backend, add ₹1000 to balance" → Backend: "Done"

REST Rules:
  POST = Create new item
  GET = Fetch/read items
  PUT = Update item
  DELETE = Remove item

Example: Getting account balance
  Frontend: GET /api/account
  Backend: Returns { accountNumber: "ACC001", balance: 50000 }

CONCEPT 2: JWT (JSON Web Tokens)
What: Secure way to prove user is logged in
How:
  1. User logs in with email/password
  2. Backend verifies, creates JWT token
  3. Frontend stores token locally
  4. Each request includes token
  5. Backend verifies token is valid
  6. If valid, processes request; if not, rejects

Token Structure:
  eyJhbGc.eyJzdWI.SflKxw...
  ↑       ↑       ↑
  Header  Payload Signature

Frontend stores in localStorage:
  localStorage.setItem('authToken', 'eyJhbGc...')

Include in every request:
  Authorization: Bearer eyJhbGc...

CONCEPT 3: Hashing (One-Way Encryption)
What: Convert data to unique fixed-length string
Why: Cannot convert back to original
Example:
  Password: "MyPassword123"
  Hash: "$2b$10$..." (60 characters)
  
  Even if hacker gets hash, cannot get original password

SHA-256 (Used in Blockchain):
  Input: "from=ACC001, to=ACC002, amount=5000"
  Output: "3e4a7c8f9f2d1b5e6c8a9d0f1e2c3b4a..."
  
  Same input ALWAYS produces same output
  Different input produces COMPLETELY different output

CONCEPT 4: TOTP (Time-based One-Time Password)
What: 2FA method that generates new codes every 30 seconds
How:
  1. Admin generates TOTP secret (string of random chars)
  2. Storage in: TOTP secret stored in database
  3. User scans QR code with authenticator app
  4. App uses secret to calculate 6-digit code
  5. Code changes every 30 seconds
  6. User enters code during login
  7. Server calculates what code should be
  8. If entered code matches, user logged in

Apps that use TOTP:
  - Google Authenticator
  - Microsoft Authenticator
  - Authy

CONCEPT 5: Mongoose (Database Modeling)
What: Library that defines database structure
Why: Prevents invalid data from entering database
Example Schema:
  const userSchema = new Schema({
    name: { type: String, required: true },
    email: { type: String, unique: true, required: true },
    age: { type: Number, min: 18, max: 120 }
  })

Validation:
  - If email empty → Rejected (required)
  - If email not unique → Rejected (unique)
  - If age < 18 → Rejected (min: 18)

CONCEPT 6: Middleware
What: Code that runs before every request
Example:
  Request comes → Middleware checks token → Controller processes → Response sent

Common Middleware:
  1. authMiddleware - Check if user logged in
  2. errorHandler - Catch and format errors
  3. logger - Record all requests

CONCEPT 7: Context API (State Management)
What: Way to share data across React components without prop drilling
Without Context:
  Component A → Component B → Component C → Component D
  (passing data through components that don't need it)

With Context:
  AuthContext stores: { user, token, login(), logout() }
  Any component can access directly:
    const { user } = useContext(AuthContext)

CONCEPT 8: Mongoose References (Relationships)
What: Link between different collections
Example: User and Account relationship
  Account Schema:
    userId: {
      type: mongoose.Schema.Types.ObjectId,
      ref: 'User'  // This account belongs to a User
    }

When querying:
  Account.findById(id).populate('userId')
  Returns: Account with full User object included

CONCEPT 9: Async/Await
What: Making code wait for long operations
Example Without Await:
  getUser() // Returns promise (not data yet)
  console.log(user) // undefined (not ready)

Example With Await:
  const user = await getUser() // Wait for response
  console.log(user) // Has data

Used for:
  - Database queries
  - API calls
  - File operations
  - Anything that takes time

CONCEPT 10: Bcrypt (Password Hashing)
What: Secure way to store passwords
Process:
  Original Password: "MyPassword123"
  ↓ (bcrypt with 10 salt rounds)
  Hash: "$2b$10$..." (cannot be reversed)
  
  Later, to verify:
  User enters: "MyPassword123"
  Bcrypt compares with hash
  If match → Password correct
  If no match → Password wrong

Salt Rounds = 10:
  - More rounds = More secure but slower
  - 10 is good balance (takes ~100ms)

EXAM QUESTION TYPES & ANSWERS:

Q: "What is the main purpose of this project?"
A: A blockchain-based banking system allowing customers to deposit, withdraw, transfer money, and manage KYC, with admin oversight, dispute resolution, and advanced features like credit scoring.

Q: "Explain blockchain in simple terms"
A: Blockchain is a chain of blocks where each block contains transaction data and a hash. Blocks are linked by referencing the previous block's hash. If anyone changes data, the hash changes, breaking the chain, so tampering is detected.

Q: "How does 2FA work in this project?"
A: Uses TOTP (Time-based One-Time Password). User gets a TOTP secret, scans QR code with authenticator app. App generates new 6-digit code every 30 seconds. During login, user enters code, server verifies, if correct, user logs in.

Q: "What are the different pages in this application?"
A: Customer pages: Login, Signup, Dashboard, Deposit, Withdraw, Transfer, Transactions History, KYC, Credit Scoring. Admin pages: Login, Dashboard, KYC Management, Deposits, Withdrawals, Transfers.

Q: "How is this different from a real blockchain?"
A: This is demo/educational. Real blockchain is distributed (many nodes), has consensus mechanism, cryptocurrency tokens. This is single node, in-memory, uses SHA-256 hashing only.

Q: "What database is used?"
A: MongoDB (NoSQL database). Stores collections like Users, Accounts, Deposits, Withdrawals, Transfers, KYC, Disputes, Suspensions, and Blockchain Blocks.

Q: "How are passwords stored?"
A: Passwords are hashed using bcrypt with 10 salt rounds. The hash cannot be converted back to original password. During login, entered password is compared with stored hash.

Q: "What are the main security features?"
A: JWT tokens for authentication, bcrypt for password hashing, 2FA with TOTP, blockchain for transaction immutability, role-based access control, account suspension for fraud.

Q: "What HTTP methods are used in APIs?"
A: POST for creating (deposits, transfers), GET for reading (accounts, history), PUT for updating (approval status), DELETE for removing.
